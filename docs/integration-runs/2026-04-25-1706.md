## Diagnostics

### diag 1 — **P0**: `run_scenario` crashes the MCP server with SIGABRT (exit 139)

**Root cause:** `src/scenario_engine/action_executor.py` calls `await self._loop.run_in_executor(None, lambda: self._cm.X(...))` for every action. The default `ThreadPoolExecutor` allocates worker threads that are **not registered with pjlib**. pjlib's `pj_thread_this()` aborts on any call from an unregistered thread:

```
python: ../src/pj/os_core_unix.c:857: pj_thread_this:
  Assertion `!"Calling pjlib from unknown/external thread.
  You must register external threads with pj_thread_register() …"' failed.
```

**Where introduced:** commit `f3e0a5b feat: scenario engine — event-driven runtime for SIP test scenarios` (the very first scenario-engine MR).

**Why per-phone MCP tools work but scenario engine doesn't:** `phone_tool_factory.py` calls `self._cm.make_call(...)` synchronously on the asyncio main thread (which is the registered pjsua thread). `action_executor.py` introduces `run_in_executor`, which moves the call to an unregistered worker.

**Fix applied (uncommitted):** added helper `ActionExecutor._run_pj(fn)` that calls `self._engine.register_current_thread()` (idempotent, gated by `libIsThreadRegistered()`) before dispatching to the executor. Replaced all 18 `await self._loop.run_in_executor(None, lambda: …)` call sites with `await self._run_pj(lambda: …)`. Verified: server stays up, B.3.1–B.3.7 all run cleanly.

**Why the test suite missed it:**
- `tests/scenario_engine/test_actions_direct.py` (and siblings) use a `MockCallManager`, not real pjsua. Mocks have no thread-affinity assertion → `run_in_executor` "works" in unit tests.
- `tests/scenarios/test_hello_world_integration.py` exercises the real engine but is marked `pytestmark = pytest.mark.integration` → skipped by default. The most recent commit `2570840 test: integration stub …` calls itself a "stub" — likely never actually executed against real pjsua.
- Net effect: classic *mocks lie*. Unit tests verified the engine's logic; the C-library interaction surface had zero coverage.

### diag 2 — **stand-side**: PortaSIP fork-and-fallback routing distorts every multi-phone test

- INVITEs to a stand-side extension are forked to **all** registered contacts of that account, including stale ones (a shadow client at `192.168.1.108:61573` was registered for `123001`, winning the answer race over our `b` phone). Symptom: `b` sees INCOMING with `last_status=100 Trying`, then the call vanishes; `a` reaches CONFIRMED with `remote_contact=192.168.1.201:5070` (PortaSIP's own).
- After ~6 s of no-answer (or another endpoint answering), PortaSIP CANCELs the unrejected legs and proxies the surviving one back to A as a 200 OK from `192.168.1.201:5070`.
- Reject-486 from B does not propagate to A because PortaSIP **falls back to another endpoint** (likely voicemail or hunt-group member) on rejection.
- Workaround used in this run: switched `b/c` to `123003/123004` (no shadow registrations); set `auto_answer=true` (or 100–200 ms answer delay in patterns) to win the race when shadow contacts exist.

### diag 3 — **stand-side**: PortaSIP returns `200 Authorization failure` for bad-password REGISTER

Instead of standard `401 Unauthorized` / `403 Forbidden`, the stand answers `SIP/2.0 200 Authorization failure` (literally — see captured response in run logs). pjsua interprets 200 as success → `is_registered` stays `true`. This makes the engine's `reg.failed` event impossible to trigger via `update_phone(password="wrong")` on this stand. Not a server-side bug; document and skip the row.

### diag 4 — **engine**: `wait-for-registration` pattern times out when phones are pre-registered

The pattern hooks on `reg.success`, but `reg.success` fires once, during `add_phone`, **before** the scenario starts. If phones are pre-provisioned in the preamble, `wait-for-registration` never sees the event → 5 s timeout. Not a P0 — every example scenario in `scenarios/examples/` uses it as the first pattern, and they all silently extend the run by 5 s × #phones. Either drop the pattern when pre-provisioned, or make it check current registration state in addition to the event.
